================================================================================
DESIGN SYSTEM & UX RULES  (MANDATORY — every UI change MUST comply)
================================================================================

These rules are NON-NEGOTIABLE. Violating any of them is a bug. When in doubt,
choose the simpler, more conventional layout over a clever one.

--------------------------------------------------------------------------------
1  VISUAL TOKENS
--------------------------------------------------------------------------------
Corner radius:
  • Buttons, cards, and dialogs → 8 px
  • Smaller controls (checkboxes, inputs, combo-boxes, tags, badges) → 4 px
  • Never use fully-rounded pills unless the element is a floating action chip.

Color system (tri-color, derived from the OS accent color):
  • Primary   — CTA buttons, active/selected states, progress indicators.
  • Secondary — secondary actions, toggle-off states, subtle highlights.
  • Tertiary  — backgrounds, dividers, disabled states, low-emphasis text.
  • NEVER introduce ad-hoc hex/rgb values. Every color must trace back to
    primary, secondary, tertiary, or a semantic alias (error, warning, success).

Typography:
  • Use the system font stack (Segoe UI on Windows). Never hardcode a web font.
  • Limit the app to 3 type sizes: Title (16 px), Body (13 px), Caption (11 px).
  • Semibold for headings and primary action labels; Regular for everything else.
  • Never use ALL-CAPS for labels longer than 2 words.

Iconography:
  • All icons must be monochrome and size-consistent (16×16 for inline, 20×20
    for toolbar, 24×24 for prominent actions).
  • Pair every icon-only button with a tooltip. No exceptions.

Spacing & layout grid:
  • Base unit = 4 px. All margins, paddings, and gaps must be multiples of 4.
  • Standard content padding inside cards/dialogs = 16 px.
  • Standard gap between sibling controls = 8 px.
  • Standard gap between logical sections = 16 px.
  • Minimum touch/click target = 32×32 px (this applies even on desktop).

--------------------------------------------------------------------------------
2  UX PRINCIPLES (follow strictly)
--------------------------------------------------------------------------------

2.1  HIERARCHY & FOCUS
  • Every view must have exactly ONE primary action. If you have two equal-weight
    buttons side by side, you have a hierarchy problem — fix it.
  • Primary action → filled/accented button.  Secondary → outlined or flat.
    Destructive → red text/outlined button, NEVER a filled red button in normal flow.
  • Group related controls visually (proximity) and separate unrelated groups
    with whitespace or a subtle divider — never a heavy rule.

2.2  FEEDBACK & RESPONSIVENESS
  • Every user-initiated action must produce visible feedback within 100 ms.
    If the operation takes longer, show a spinner or progress bar immediately.
  • Disable the trigger control during async work and re-enable on completion.
  • Toast notifications for success/info, inline error messages for validation.
    Never use a modal dialog for success messages.
  • Hover states on every interactive element (buttons, cards, links, tabs).
  • Pressed/active state must be visually distinct from hover.

2.3  CONSISTENCY
  • Identical actions must look and behave identically everywhere in the app.
  • Control order in dialogs: fields top-to-bottom in reading order, then
    action buttons at the bottom-right, with primary action rightmost.
  • "Cancel" / "Close" is always to the LEFT of the primary action button.
  • Use consistent verb-style labels: "Save", "Delete", "Apply" — not
    "OK", "Yes", "Confirm" (unless it is a destructive confirmation).

2.4  PROGRESSIVE DISCLOSURE
  • Show only what the user needs right now. Collapse advanced options behind
    an expander or "Advanced…" link.
  • Tooltips for power-user shortcuts; never rely on tooltips for essential info.

2.5  ERROR PREVENTION & RECOVERY
  • Prefer constrained inputs (combo-boxes, sliders) over free-text where the
    set of valid values is known.
  • Destructive actions require a confirmation dialog that names what will be
    destroyed: "Delete 3 images?" — never just "Are you sure?".
  • Allow undo wherever possible instead of pre-confirming.

2.6  ACCESSIBILITY BASELINE
  • Minimum contrast ratio 4.5:1 for body text, 3:1 for large text and icons.
  • All controls must be keyboard-navigable (Tab order, Enter/Space to activate).
  • Focus ring must be visible and must not be suppressed.

--------------------------------------------------------------------------------
3  COMMON UX PATTERNS (use these; do NOT invent novel alternatives)
--------------------------------------------------------------------------------

Cards (image cards, content cards):
  • Fixed aspect ratio or fixed height within a grid — never allow cards to
    have wildly different heights in the same row.
  • Hover: subtle elevation increase (shadow) + optional overlay.
  • Selection: accented border or check-mark overlay — never change the card
    background to accent color (it kills readability).
  • Context menu on right-click, with the most common action also available
    via a single primary click or a visible icon button.

Dialogs / Modal windows:
  • Title bar with concise noun phrase ("Export Settings", not "Configure").
  • Content area with logical field grouping.
  • Bottom action bar, right-aligned: [Cancel] [Primary Action].
  • Dialogs must not exceed 480 px wide unless they contain a preview pane.
  • Always closable via Escape key.

Tab bars:
  • Tabs must never wrap to a second line. If there are too many tabs,
    use a scrollable tab strip or an overflow menu.
  • Active tab indicator = 2 px bottom border in primary color.
  • Tab label = single word or two words max. No icons-only tabs without
    an adjacent text label.

Toolbars / action bars:
  • Group related actions with 8 px gaps; separate groups with a 1 px vertical
    divider or 16 px gap.
  • Max 7±2 visible actions; overflow the rest into a "⋯" menu.
  • Place the toolbar at the top of its context (window or panel), never
    floating ambiguously.

Scroll areas:
  • Scroll bar must be visible on hover (never permanently hidden).
  • Content must never be clipped without a scroll indicator.
  • If the scroll area can be empty, show an empty-state message centered
    in the area — never leave a blank void.

Forms / settings:
  • Label above or to the left of the control — never below or to the right.
  • Align labels and controls on a consistent vertical axis.
  • Real-time validation: show error messages below the offending field as
    soon as focus leaves the field.
  • Disable the "Save" / "Apply" button until the form is dirty.

Progress indicators:
  • Determinate (progress bar with %) for operations where duration is known.
  • Indeterminate (spinner or pulsing bar) only when duration is unknown.
  • Always show a text label next to the indicator describing the operation.

Toast / snackbar notifications:
  • Appear from the bottom or bottom-right, auto-dismiss in 3–5 seconds.
  • Provide an explicit dismiss button ("×") for users who want it gone sooner.
  • Never stack more than 2 toasts; queue additional ones.

Context menus:
  • Keep entries ≤ 10. Group with separators if > 5.
  • Destructive items at the bottom, visually differentiated (red text).
  • Include keyboard shortcut hints right-aligned.

Drag-and-drop:
  • Show a clear drop target highlight (dashed border or tinted overlay).
  • Provide a non-drag alternative for every drag interaction (button, menu).

--------------------------------------------------------------------------------
4  STRICT LAYOUT VALIDATION CHECKLIST  (run BEFORE finalizing ANY UI change)
--------------------------------------------------------------------------------

You MUST mentally execute every check below. If any check fails, fix it before
presenting the code. Treat a failed check as a blocking bug.

  □  OVERFLOW CHECK
     – Does any text truncate or overflow its container at default window size?
     – Does any widget extend beyond its parent's bounds?
     – At minimum window size (the resize limit), does everything still fit?
     Fix: use elision (QLabel word-wrap or elide-right), scroll areas, or
     reduce content. Never let content silently disappear.

  □  ALIGNMENT CHECK
     – Are all labels and controls aligned on a grid (multiples of 4 px)?
     – In side-by-side layouts, are baselines of text labels vertically aligned?
     – In dialogs, is the button row right-aligned and consistently spaced?
     Fix: use QFormLayout for label-field pairs, addStretch() for right-
     alignment, consistent spacing constants.

  □  SPACING CHECK
     – Is there a consistent gap between every pair of adjacent elements?
     – Is there no case where two elements are touching (0 px gap)?
     – Are section gaps larger than intra-section gaps (hierarchy of space)?
     Fix: set layout spacing and contentsMargins explicitly. Never rely on
     default margins.

  □  HIERARCHY CHECK
     – Is there exactly one primary action per view/dialog?
     – Is the visual weight of each element proportional to its importance?
     – Does the user's eye naturally land on the most important element first?
     Fix: adjust font weight, button style, size, or position.

  □  EMPTY STATE CHECK
     – What does this view look like with zero data? Is there an empty-state
       message or illustration?
     Fix: add centered placeholder text/icon for every list/grid/scroll area.

  □  REDUNDANCY CHECK
     – Can this layout be simplified? Are there unnecessary nested containers,
       redundant labels, or duplicate controls?
     – Is there a simpler, more conventional layout that achieves the same goal?
     Fix: flatten nesting, remove decorative-only wrappers, merge controls
     that do the same thing.

  □  CONSISTENCY CHECK
     – Does this new/changed UI element match the style of existing identical
       elements elsewhere in the app?
     – Are fonts, colors, corner radii, and spacings reusing the design tokens
       defined above — not introducing new magic numbers?
     Fix: refactor to use shared constants or widget subclasses.

  □  INTERACTION CHECK
     – Does every button/link/card have hover, pressed, disabled, and focused
       states?
     – Does keyboard navigation (Tab, Enter, Escape) work logically?
     Fix: add missing states. Check setFocusPolicy and event handlers.

  □  SUPERIORITY CHECK  (critical — apply last)
     – Step back and look at the layout as a whole. Is there a simpler or more
       standard way to present this information?
     – Would a flat list be better than a grid? A combo-box better than radio
       buttons? Inline editing better than a dialog?
     – If the answer is yes, refactor to the simpler layout BEFORE shipping.
     Rule of thumb: fewer containers, fewer clicks, fewer visual elements =
     better. Always prefer the layout with less nesting and fewer moving parts.
